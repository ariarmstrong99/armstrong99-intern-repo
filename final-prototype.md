What are best practices for testing high-fidelity prototypes? 

The goal of testing a high fidelity prototype is to determine if users are successfully their tasks properly. The testing should be focusing on concrete scenarios that test for usability issues based on user goals rather than on subjective opinions from users on the high fidelity prototype. 
Designers should also encourage participants to verbalise their thought process as they conduct the testing and interact with the high fidelity prototype. 
It is also important to not respond to users with answers to help them, as when they they are confused, hesitate or can’t understand something. 
Many designers also give out subtle physical Cues that indicate the user is doing something the wrong way, this needs to be avoided as well.  
The testing findings should also be evaluated for certain patterns across multiple participants. If multiple users are experiencing the same problems, this means that there is a strong possibility of a usability problem.


How do you analyze feedback from stakeholders vs. end-users? 

Feedback from stakeholders and users is valuable. Users can determine if a design is usable and if it manages to actually solve real problems faced by the users. 
When going through feedback, designer should be focusing on certain behaviours, pain points, if users are managing to complete their tasksl any confusion they experience, and if errors are being made. 
If more than user is experiencing the same or similar issues, then it becomes likely there is an underlying design issue that needs to be fixed. 
Stakeholder feedback usually helps a designer align the design with business goals, product goals, and  any constraints that exist. When user feedback and feedback from key stakeholders is in conflict, UX designers should be investigating the underlying reasons instead of focusing on one over the other. 
For example, a product manager might ask the designer to add a new feature to the product in order to attract more users, whereas user testing results indicate that users  are finding several existing features to be confusing in nature and struggle to use them properly. 
In a situation like this the designer would aim to present the key stakeholders with research findings to find a solution that helps meets business and user goals 



What techniques help designers prioritize feedback and avoid overdesigning? 

Overdesigning can be avoided by prioritising feedback based on n how much it impacts users and business goals. It is not possible to implement every suggestion. 
It is always useful to look at patterns instead of individualised feedback. If a user mentions an issue, it could be worth investigating. 
If more than five users are experiencing the same problem, it is likely that this would need attention. Designers should also be focusing on feedback that is related to the successful completion of tasks by users and any usability issues experienced by users over superficial aesthetic changes. 
If users are struggling to locate the primary button on a screen due to the button being placed on different positions on each screen, it is pointless to try and improve the aesthetics without fixing this issue first. 
All feedback needs to be evaluated from the lens of whether it is going to help users complete their tasks more efficiently. 


How should usability testing findings influence prototype iterations? 

Any iterations that need to be made should be based on usability test findings. It is always useful to look for patterns. 
If more than one user is experiencing the same problems, it is an indication of an underlying design problem. It is not possible to change the designs based on each and every individual comment made by the users. 
It is much better to focus on patterns and address issues that affect multiple users. 


How can UX designers balance design vision vs. user feedback when making iterations? 

A designer needs to investigate what exactly a user is after. The ultimate goal of a designer is to use user feedback in order to understand the problems that are being faced by users and then determine the best solution. 
Users should not be relied on for design solutions. For example, it is okay to observe users and see where they experience issues, become confused, and fail to complete tasks during user testing. 
But asking them how a design can be improved is not helpful since users do not always understand the underlying issues in the design. 
Designers should be investing their time investigating why a user does things a certain way instead of relying on their design vision. 
It is always good to have a gut feeling and design vision, but if a designer is unable to let go of this when confronted with evidence during the research phase, then it becomes problematic. 


If conflicting feedback arises, how would you decide what to change? 

Receiving feedback that is conflicting is quite common in UX. A designer should aim to Analyse the feedback they receive with user needs, and business goals. 
It is always important to understand why the feedback is being given in the first place. It is also important to then look at the evidence. If there is evidence indicating that the same issue is being experienced by multiple users, this needs to resolved. 
Also, if users are finding it difficult to complete key tasks efficiently, then resolving this is more important than focusing on improving the visual aesthetics of the app. 
Changes need to be made in order to impact to user experience and business goals. If a user cannot even complete the onboarding process successfully and abandon the process out of frustration, then resolving this should take precedence over any other minor visual tweaks to the interface of the app.


What are the risks of iterating too much without clear direction? 

Any interactions that take place need to occur after analysing evidence from user research.  Usability test findings, user interviews, surveys and business goals all need to be accounted for when making any iterations. Without looking at evidence and iterating, there is a risk of being involved in a long process of simply tweaking screens constantly and responding to feedback without actually resolving any key issues identified during the research phase. 
For example, a key stakeholder might ask to make certain changes to the onboarding process, but if evidence from  research phase is not looked at, then the onboarding process will look different, but fail to solve any usability issues, fail to meet user needs or help the business achieve its goals.
