What are the core features of Figma, and how is it different from other design tools? 

Figma is the gold standard software in the world of UX design for the purpose of creating UI and for prototyping. 
There are other platforms such as AXURE, but figma is by far the best because it is versatile and can be used to create wireframes, design systems, high fidelity UI and prototypes. 
The core features of the platform include the ability to conduct a developer handoff in dev mode, prototyping and being able to create design libraries. 
Figma is unique because you can collaborate with your team in real time. Multiple team members can join the same design file and be able to work on it simulatneously.  
This is not something that can be done in competitor platforms such as sketch (only for Mac OS) and adobe XD (now discontinued). 
Figma is also cloud based, which means the experience is exactly the same across different operating systems. 
Figma has now also started to add A.I features such as figma make, which make it a very powerful tool.



How does Figma support collaboration between designers, developers, and PMs? 

Multiple team members can collaborate in real time and work on the same design file. 
A designer can continue to work on the wireframes and UI, whilst a product manager can review each individual flow and leave comments for the designers. 
A developer can access the dev mode feature to have a look at measurements, spacing, Color, typography and conduct a quick QA, which can then be cross checked by the designer. 
All of this can occur in real time and simultaneously without any interruptions for each respective aforementioned party. 


What are components, variants, and auto-layout, and how do they improve design efficiency? 


A component can be conceptualised as a reusable UI element. For example, a button. Components have a main component and also have instances. 
A main component is something that if you update, will update all the other “instances” of that component automatically. This saves a significant amount of time for designers as they work on increasingly complex designs. 
In contrast, a variant is a different state or version of a component that is grouped together. For example, a hover state of a button is a variant. 
A loading state is another variant for a button. Variants make it much more easier to manage components and they also allow designers to work from one source. 
Auto layout on the other hand is a feature that is responsible for controlling spacing, alignment, and resizing. Auto layout ensures that as content changes, the frames automatically adjust without manual repositioning. 
Variants, auto layout and components all make design process much more faster by spending up repetitive work and allow designers to build much quicker.


How do you share and hand off designs to developers using Figma? 

It is very easy to share designs in figma. People can be addded to the file via their email account or by sending them a link to access the design file directly. 
Developers are able to use dev mode and inspect spacing, colours, typography etc. 
Designers are able to provide developers components that are well organised, prototypes and help establish how screens are meant to behave. 


How would you organize a design system in Figma to ensure consistency? 

A design system can be organised into clear sections so that team members are able to easily find and reuse the proper asssets.
I would separate each section by page in a figma design file The first page would contain information about the foundation of the system, such as typography, colours, spacing, and icons. 
The next section would then focus on reusable components such as buttons, form fields, navigation elements and different types of cards. I would also create a specific section for common patterns that are used. 
This can be a combination of what specific components are used together, such as forms, modals, and other dashboard layouts. The very last section would be templates and this will show the entire screen or page structure. 


If a developer needs details about a design component, how would you provide that using figma?

If a developer requested me for more information about a specific design component, I would simply ask them to open that particular component in dev mode. 
When in dev mode, they are able to properly inspect key information like spacing, colours, typography, and variables. 
When a component has specific behaviours such as hover states, I would provide the developer with an prototype and also add comments that would help them understand how the prototype worked. 

What challenges can arise when collaborating in figma and how can you avoid them?

When multiple people are collaborating on the same file in figma, there is a chance that the design file can become quite messy, cluttered and disorganised. 
This can easily occur if components are named haphazardly and randomly or have similar names. 
It can also occur when there is no clear structure to the file and people begin to clutter the file with assets, designs, and textual Information and all without separating them Properly. 
Another key issue is that when lack of proper documentation is kept on when designs were updated and why, it can lead to miscommunication because people might not necessarily understand why certain changes were made. 
A lot of these issues can be mitigated by ensuring each figma file has a clear structure, is well organised into different sections and there is a shared design system with proper naming conventions for components and variants that do not lead to confusion. 
It is also important to set the right permissions for each team member that has access to the file because many times members might have access the file but are restricted by permissions. This can also slow down collaboration and lead to issues.
