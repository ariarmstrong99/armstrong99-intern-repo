What are best practices for conducting user interviews? 

When conducting user interviews, the aim is try and gather information that is unbiased, accurate and honest. The purpose is to understand a user and their experiences. 
Therefore, it is never a good idea to ask leading questions such as “did you like the onboarding flow?”. Because asking leading questions can make a user think that a certain type of answer is expected from their side. 
It is also important to establish from the very beginning of the user interview with the users involved that you are not looking for certain type of answers. 
Futhermore, it is also helpful to establish with users prior to commencement of user interviews that there is no wrong answer and that you are merely trying to get them to respond honestly. 
This makes users feel more comfortable giving authentic responses without trying to please the interviewer with curated responses. 
It is also important to establish rapport with the users prior to commencement of the user interviews. 
This helps establish trust with the users, makes the experience feel more human and less transactional in nature. 
It also allows the users to be more vulnerable when rapport is built correctly. This in turn can lead to responses that provide deeper insights into user behaviour and thinking. The idea is to have user interviews resemble a relaxed dialogue between two people. 
Listening to users as they talk, not interrupting them or talking over them is also important. The interviewer should always let the user talk first and then ask follow up questions if they need something to be elaborated or clarified.


How do you run an effective usability test without influencing the user? 

Running a user testing session that does not influence the user is very important. Because if a user is influenced by the tester, then the value provided by the user testing session becomes limited. Often times during a user testing session when a user struggles to complete a scenario, testers might give out subtle cues that could provide users with a hint if they are doing things correctly. 
Another issue is that sometimes testers might explicitly tell the users that “‘you are doing this wrong”. But this prevents the collection of valuable insights during the session. 
If a user struggles with something, the tester should not say anything or give any cues. If asked by the user “am I doing this correctly?”  the testers should remain calm and respond with “‘there are no right or wrong answers here, please do what you feel is right according to you”. 
This allows the testing session to capture where the user struggles or becomes confused. This provides the testers with valuable data that can pinpoint underlying issues with the products interface. 
The tester should also be focused on observing the user instead of trying to teach the user how to do things during the session. 
Each user is going to respond differently during a testing session and the tester should focus on understanding how the users interact with the products  interface.u


 
What types of questions should UX designers ask during interviews? 

During user interviewers, interviewers can ask users questions that help them gain understanding into a users behaviour, goals and motivations. 
The focus should not be on proving the design direction they have taken or planning to take as the correct option. 
A good user interview often entails  attempting to explore the users behaviour. For example, if the aim is to improve aspects of the focus bear app, it might be helpful to users questions such as “‘can you please tell me how you conduct your study sessions. 
I want to understand the steps you take to organise your sessions”. These types of questions give us insight into the user behaviour without asking leading questions. 
Asking these types of questions basically helps the interviewer get insights and understand more about a users behaviour in a non biased manner. 


How can recording and note-taking improve research accuracy? 


Due to the technological advances in contemporary society, UX designers are now better able to conduct user research without having to rely on their memory. 
UX designers are able to record user testing sessions with software such as loom. User testing sessions are often between 30-90 minutes long. 
In the past, a scribe would be present to record user reactions prior to the advent of reliable video recording and transcription services. 
However, now we can record user testing sessions with relative ease and observe what a user does, says, and where exactly they experienced difficulties or were confused. 
This can easily be accomplished by simply replaying the user testing recording to go over details that the tester cannot remember. 
Taking notes is important as well since it helps UX researchers identify behaviours, and other issues that were observed during a user testing session. 
Making clear, succinct and well organised notes is fundamental to UX research. 
Notes can be very helpful for a UX researcher because a recording might show a user getting confused, but they do not allow for the UX researcher to write down in real time, why they think this is the case. 
For example, a user might struggle during a particular step during user testing or be confused by a feature. 
A UX researcher can write down on paper “struggled significantly on scenario number two”. The UX researcher might forget where exactly the user struggled and during what step if they did not make this note. 
Since many users are involved in user testing and because it is not always possible for a UX researcher to remember all the details for each users and session, notes can be a valuable supplement to session recordings. 


What are common mistakes to avoid when conducting user research? 

One of the most common issues and mistakes that can occur during user research is influencing users inadvertently. 
Another problem is trying to validate an idea or design direction via the user research instead of trying to learn from user behaviour. 
It can also be quite problematic when UX researchers try to help users during UX research. 
This is particularly relevant for user testing when a UX researcher might accidentally point out to the user that “they are not doing something correctly” either subtle cues given out by their body language or by explicitly telling users what to focus on during the testing. 
Similarly, when UX researchers focus on what a user says instead of what they are doing. 
For example, a user might say during a user testing session that they found the app easy to use. 
However, it might be more palpable by going through the session recording that the user did hesitate multiple times during the session, was confused on numerous occasions and took longer to complete certain tasks. 
A competent UX researcher would probe further here to identify the underlying reasons instead of relying entirely on the users verbal statements. 
Sometimes when UX research is conducted with few users, it is not enough to rely upon. 
Many UX researches make the mistake of making conclusions from few participants instead of trying to find patterns across more users.  
The biggest mistake that UX researchers make is entirely ignoring any feedback that goes against their beliefs or assumptions even in the presence of contradictory evidence. 

If a user struggles with a feature during testing, how do you know if it’s a UX problem or a learning curve issue? 

A good product does not require a significant learning curve for new users. A good product makes sure that users can learn how to use it rather quickly and intuitively. 
During feature testing, if multiple users are having the same issues, it can point towards this being a UX problem. 
For example, if a user is struggling to set up their blocking schedule on focus bear app and many other users have the Same issue, then it is most likely a UX problem. 
This is because it would indicate that the user interface is not intuitive enough for users to understand how to accomplish the aforementioned task. 
Also, if users expect the interface to behave in a certain way, and it does not tend to behave in this manner, then it means the product does not meet the mental model of the users.  
Another way to determine if the issue experience is a UX problem or learning curve issue is by seeing how quickly users can recover. 
When users spend time exploring the app, this is normal but if it takes them longer to recover due to constant backtracking, hesitation, confusion and are unable to complete the task, then this is clearly a UX failure.



What’s the best way to document usability testing results so they are useful for design decisions? 

The best way to document usability testing results is by focusing on the findings of the session. It is not suitable to simply state what a user has said during the session or highlight how they behaved and/or interacted with the products interface. 
The usability testing report should clearly highlight a hypothesis, what the users were trying to accomplish, each individual scenario that was given to the users, and success criteria for each scenario. 
The report should also discuss the exact areas where a user experienced issues, why they experienced those particular issues, and what this means for the products design. 
For example, if multiple users were failing during user testing to successfully create a blocking schedule and were abandoning the task, then this is an important finding. 
The important finding is that the user interface is not intuitive enough and fails to provide enough guidance to the user on what steps to take in order to successfully a blocking schedule. 
A good usability testing report should aim to focus on highlighting patterns across multiple users, instead of focusing more on Individual results.  
Additionally, when findings are highlighted, they should be linked to a specific design decision or recommendation to fix the issues that were identified and how it impacts the user experience. 
Having good clarity with documentation will ultimately make it easier to fix the issues identified in the user testing sessions.
