What is the difference between a static wireframe and an interactive prototype? 


A static wireframe is simply a screen that shows how it will structured. In contrast, an interactive prototype demonstrates how a user will navigate through the flow and interact with the screens.  
For example, a static wireframe for focus bear’s redesigned onboarding flow might show each individual screen clearly, but it will not allow a user to interact with the interface in any way shape or form. 
In addition, users won’t be able to navigate from one screen to another either for this particular wireframe flow. In contrast, an interactive prototype would allow the user to navigate through the entire onboarding flow by clicking on a screen and then being able to interact with the interface as if they were using the focus bear application. 
A wireframe will allow a designer to think about how a screen can be structured. In contrast, a prototype allows designers to actually test user behaviour and navigation.



How does prototyping help validate usability before development? 

Prototyping is very useful in helping validate usability prior to proceeding with development because it allows designers to observe and test how a user will interact with the interface. 
A designer can articulate how a feature will work, but it is entirely different when a user is able to interact with an interface and a designer can test out the interface with a user as they click through screens, complete certain tasks and actually go through the flow like it was the real product. 
In this manner usability issues can easily be identified when they are significantly easier to fix. 
For example, for the redesigned focus bear onboarding flow, the design team can create a clickable prototype and get users to go through this onboarding flow. 
During the user testing sessions, it may be discovered the users are finding the redesigned onboarding quite confusing in nature and abandoning the process altogether between screen 3-5. 
The design team can then proceed to rectify these issues prior to the commencement of development process.


What are the key features of Figma’s prototyping tools (e.g., linking screens, hover states, transitions)? 

The prototyping tools in figma allow designers to actually simulate how an interface will behave prior to the completion of the development process. 
Figma allows designers to use the screen linking feature to link screens together so that when they click a button, card or link, it helps the user navigate to another screen.  
There is also support in figma for interaction triggers. Interaction triggers react to actions like pressing a button, and being able to drag certain objects. 
This feature is intended to allow designers to visualise interactions that are realistic in nature. Figma also has support for transitions and animations where designers are able to determine how a user will transition between screens using states such as smart animations and dissolves. 
These type of transitions help indicate to the user that there is a change in state that is occurring and this in turn helps create a better user experience. Figma also lets designers create interactive components whereby a single component can change based on states. 
For example, buttons usually have multiple states, such  as hover, disabled and pressed. 



How do designers use prototypes for usability testing and stakeholder feedback? 

Prototypes are often used in usability testing to allow users to experience a design in the most realistic manner possible. A prototype is basically given to users during the usability testing session. 
Scenarios are also given during this testing session. Users then basically complete the scenarios/and or tasks given out to them during the session by interacting with the prototype. 
Designers observe what a user does, and says during the session with special emphasis on areas where they become confused, hesitate or fail to complete a certain task/ and or scenario. 
A prototype can also be used for feedback from key stakeholders such as product a manager, and developers. These stakeholders can understand how a proposed solution tends to work as a cohesive experience. 
Stakeholders can click through the prototype and then evaluate whether it managed to meet users needs, and business goals. After goingt through the prototype, the product manager might say that a certain feature involves too many steps, whereas a developer might indicate that it will take too long to develop the feature in its current form. 
A designer can then make changes to the design based on their feedback. This process becomes much harder when using static wireframes because it’s difficult to visualise how the interface responds for key stakeholders when there is no possibility of being able to navigate from one screen to another. 

What usability issues can a clickable prototype reveal that a static wireframe cannot?

A static wireframes has significant limitations in contrast to a clickable prototype. A clickable prototype allows users to interact with the interface freely and it becomes possible to identify usability issues related to navigation. 
For example, a wireframe of redesigned onboarding process for focus bear might be appearing fine as a static wireframe. But once users interact with the clickable prototype, it becomes very clear that users do not know what particular buttons to click on and tend to get confused between the onboarding steps or simply abandon the process because it is too frustrating. 
It is impossible to identify these issues on a static wireframe. Clickable prototypes also allow designers to understand what users expect from certain elements on the user interface. A user might expect certain parts of the interface to be interactive and try clicking those to receive a response. This is not possible to identify on a static wireframe. 


How can prototyping help designers communicate ideas more clearly to developers? 

A developer may understand a design to a certain extent by looking at static wireframes. However, they cannot understand how it will behave when users interact with the interface. A prototype makes things more clear because they can see can navigate the user flow, screen transitions, interact with buttons and see error states. 
By going through the actual prototype, they can see how the interface behaves. The developer can go through the entire user journey by interacting with the clickable prototype, something that is not possible with a static wireframe. 
In this manner, a developer can identify if there will be technical constraints in developing the prototype. The developer can explain this to the designer if this is the case and the designer can then take the feedback on in order to make changes. 


What are common mistakes designers make when creating prototypes? 

A prototype is meant to be a tool for testing rather than being a visual representation of a design. Many designers tend to spend a significant amount of time and energy treating a prototype like a design artefact that needs to be constantly polished rather than treating it like a testing tool that allows designers to validate ideas, test for usability issues and communicate with key stakeholder. 
In this instance, the prototype is “heavy” with all types of animations and visuals that make the prototype look good, but it fails in its most fundamental purpose. The other issue is that designers make the prototype too restrictive and the users have to follow the exact path the designer wants. 
This removes the freedom of the user to explore and also removed the possibility of finding usability issues that tend to emerge by deviating from the laid out path. Users need to be given the chance to make mistakes. Try on different paths, and interact with the prototype as they need.
