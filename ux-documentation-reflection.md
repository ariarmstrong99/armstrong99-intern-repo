
What types of UX documentation are usually created in google docs?

In product design and UX design, google docs is often utilised for various purpose. Google docs is quite versatile and easy to use, often make it a great platform in addition to miro.
1. Research documentation:  interview guides, usability test plans and reports, findings of surveys, summaries of UX audits and competitor analysis. Google docs are also quite useful to record how certain research decisions were made, what particular lessons were learnt during the process and what issues were identified
2. Personas, journey maps, empathy maps and value proposition canvas:  Any research that is conducted on these particular deliverables can recorded in google docs. For example, user research pertaining to the mental models of users, their cognitive patterns, their pain points and goals can be recorded in google docs and shared within in an organisation. This allows multiple departments of the same organisation to have a shared understanding of the user research. This can then be utilised in collaboration sessions where journey mapping can take place as a team.


3. Design rationale: this type of documentation is quite crucial in UX and product design. Detailed records of why certain decisions were made during the scope of a project or sprint. This is important because it allows designers to track their decisions with more ease.

4. Meeting notes: meetings can be summarised in google docs for team members to view. This in important because it becomes significantly easier for team members to view meeting summaries. This can be quite useful for summaries for intense brainstorming sessions and other workshops

5. Handoff documentation: this type of documentation is very important, particularly in areas like fintech and enterprise UX. 



How should design decisions and rationale be documented for future reference?


All decision decisions need to be documented in a manner that is able to clearly articulate why each individual decision as made. Additionally, compelling evidence needs to be provided to explain how it supports the rationale for the decision being made in the first place. 
Furthermore, the type of outcome that is expected by each design decision being implemented for users and for the businesses must also be noted.  A key issue in workplaces is that many designers often fail to provide adequate rationale for many outcomes. 
This often occurs due to oversight or time constraints. 
For any design decision and rationale documentation to be considered robust, it needs to fully explain the entire sequence of reasoning that culminated in the final decision being made. 
For example, if a certain aspect of an app was redesigned by a product designer, then the design rationale documentation must detail information about why this occurred.
The designer must note down that “‘user testing results indicated that many users found the feature or part of the app confusing and this ultimately had an adverse impact on their ability to perform certain key tasks”. 
Evidence of the user testing results summary can be included in the design rationale, alongside evidence of the users mental models and how the current state (‘prior to redesign) of the specific feature or part of the app that was redesigned based on conflict with user mental models.
Documenting this process is very important because in organisations, staff members tend to change over a period of time. 
New staff members need to understand clearly why certain decisions were made. If this is not done properly, then there is a risk of the same mistake being made repeatedly in the future due to lack of proper record keeping and documentation of design decision rationale.


What is the best way to structure user research findings to make them easy to read?

UX user research findings must be presented in a manner that shows the methodology, specific research objectives, the key findings, evidence that supports the findings and next steps that have been recommended. 
This information needs to be presented in a clear manner that makes it easier t synthesise the aforementioned information and action it appropriately.
It is recommended to implement clear headings, and utilise visual hierarchy to make it easier for the reader to identify key findings when they scan the document quickly. 



How can UX documentation be used to support accessibility and inclusive design?

Accessibility and designing for access is a fundamental part of UX design. UX documentation can be specifically created for the purpose of recording acessibility requirements for making designs more usable and accessible., this type of UX documentation must properly specify specific the following:

typography requirements to ensure that all text is legible across all viewports, including for individuals with Color blindness 
Color requirements 
Differential types of captioning for media
Support for keyboard navigation
Screen reader compatibility 

When these aforementioned aspects are recorded in specific UX documentation, it helps us as designers ensure that there is a degree of consistency in terms of accessibility amongst various stakeholders that are involved in implementing the designs.



If you conducted a user testing session, how would you document the results?

If I was Involved in a user testing session the results would be documented in a manner that would capture what the users did, but also how and why they experienced certain difficulties during the scenarios for the testing session and how this would impact the product and it’s user experience. 
I would outline each individual scenario in depth, the hypothesis for each scenario would also be listed, the number of participants, duration of each testing session and the comments made by each user would all be documented in the report. 
Documenting key user statements made during the testing session  is important because it shows us how users perceived the user interface and it reveals to us their mental models in more depth. 
The findings from all user testing sessions would also be synthesised appropriately into key themes that can be identified from the results of the user testing sessions.
In addition, any issues that were identified during the session would be explained via idenfifying the root cause of the issue and ultimately a recommendation would also be listed to resolve the issue that was identified. 
The aforementioned issues would be ranked based on their severity as well. Links to the user testing video session would also be listed here, alongside screenshots. 
Documenting things in such a manner would help the design team prioritise which issues need to resolved first and allow the team to make more informed decisions.


How can a well structured UX document help or prevent miscommunication between teams?

well structured UX documentation is essential in order to prevent any type of miscommunication or misunderstanding between different team members. This is accomplished by having a common understanding of the problem, research, test results,and  design decisions. 
A shared UX document can be viewed by developers, product manager, project manager and designers.
In this manner the entire process becomes much more streamlined instead of these aforementioned team members having to engage in a back in forth process of information conversations in order to get a better understanding of why certain design decisions were made.
For example, during a user testing session a user might struggle to progress from one screen to another has they go through a scenario for an app. The shared User testing document can be extremely helpful that would provide more context to members of the team, such as project manager and product manager to see how fixing the issue identified can impact the business goals and also user goals.

What challenges might arise when writing UX documentation, and how can you avoid them?
One of the key issues that arise when writing UX documentation is being overly detailed and complex or being too brief.
When documentation is vague, ambiguous, and has poorly defined requirements, this can create issues. When UX documentation is not update properly overtime, it can lead to inconsistent implementation of design. 
It can also cause miscommunication or lead to communication breakdowns when accessibility requirements are recorded poorly, defined poorly and appear vague. 
In order to avoid such challenges, it is best to be clear, concise, well structured and organise UX documentation in a manner that has clear headings with proper visual hierarchy that is able to explain what decisions were made when, why and how were they implemented.
It is important to review and update UX documentation regularly so as to ensure it is accurate and still relevant.
